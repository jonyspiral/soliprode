# Auditoría UI SoliProde 2026-06-02

## Objetivo

Cruzar la UI actual de SoliProde contra estos contratos:

- `src/content/docs/contracts/voice-contract.md`
- `src/content/docs/contracts/athletic-social-design-contract.md`
- `src/content/docs/contracts/ui-ux-implementation-contract.md`

Esta auditoría no corrige todavía. Ordena el mapa de riesgo antes del refactor visual.

## Metodología

- Render real revisado en local con Playwright:
  - `/`
  - `/login`
  - `/register`
  - `/matches`
  - `/rankings`
  - `/dashboard` redirige a `/login?next=%2Fdashboard`
  - `/admin` redirige a `/login?next=/admin`
- Viewports auditados:
  - desktop `1440x1100`
  - mobile `390x844`
- Lectura de código para estados no visibles sin sesión real:
  - `src/app/dashboard/page.tsx`

## Resumen ejecutivo

- Home, Login y Register ya entraron más en carril `Athletic Social`, pero todavía tienen deuda menor de jerarquía y escalado de bloques secundarios.
- Matches y Rankings son hoy las pantallas con más drift funcional-visual: demasiado estado explicativo, poco juego primero y demasiado espacio dedicado a informar antes de empujar una acción.
- Dashboard sigue siendo el punto con mayor riesgo de tono administrativo y métricas falsas por código, aunque no se auditó con sesión viva en esta pasada.
- Admin debe seguir aislado del lenguaje del jugador, pero hoy todavía comparte demasiados patrones visuales del producto principal.
- No existe ruta `/profile` separada; la cuenta hoy vive repartida entre shell y dashboard.

## Hallazgos concretos

### Home

Estado:

- Respeta mejor el tono y el foco principal.
- Mobile y desktop ya no están rotos.

Hallazgos:

- El bloque `Inscripción inicial` sigue teniendo demasiado peso debajo del hero.
- Mercado Pago ya bajó a badge, pero todavía genera warning visual/técnico por escalado de asset.
- La solidaridad está mejor ubicada, aunque el bloque de apoyo puede volver a crecer si no se controla.

Decisión:

- No urgente.
- Ajuste posterior de compactación, no rediseño.

### Login

Estado:

- Muy alineado con el contrato de voz.
- CTA principal claro.
- Buena proporción hero/card.

Hallazgos:

- La pantalla depende de no sumar más explicación o notices.
- Si OAuth agrega errores persistentes o banners, puede romper rápido la regla de una pantalla, una acción principal.

Decisión:

- Pantalla estable.
- Solo vigilar que futuros errores no inflen la UI.

### Register

Estado:

- Tono correcto.
- Hero y CTA claros.

Hallazgos:

- El stepper ocupa espacio, pero todavía no destruye foco.
- El bloque de `Código de promotor` más la caja de “Grupo y equipo de 11” pueden volver la pantalla demasiado larga en mobile.
- El formulario ya está en el límite de densidad aceptable para mobile.

Decisión:

- Mantener.
- Prioridad media en simplificación futura.

### Dashboard

Estado:

- No auditado con sesión viva en esta pasada.
- Se revisó por código.

Hallazgos P0:

- Usa métricas y visualizaciones placeholder no confirmadas:
  - `Cómo viene tu torneo`
  - `Racha actual`
  - `Próximos partidos` mockeado
- Mezcla juego, cuenta y administración personal sin una jerarquía clara.
- Copy todavía cae en expresiones como `participación activa`, `falta activar tu participación`, `estado real de tu cuenta dentro del torneo`.
- Hay demasiadas cards para una sola pantalla y varias compiten entre sí.

Decisión:

- Primer candidato real a refactor visual posterior.
- Necesita pasar a “juego primero, explicación después”.

### Matches

Estado:

- En estado vacío renderiza bien, pero no se siente divertido.

Hallazgos P0:

- Hay dos notices seguidos antes de cualquier acción.
- Los tres stats `Abiertos / En vivo / Cierran hoy` aportan poco cuando todo vale `0`.
- La pantalla vacía prioriza explicar el fixture antes que empujar a crear cuenta o volver después.
- En mobile, las stats vacías ocupan demasiado espacio por el valor informativo que tienen.

Decisión:

- Debe simplificarse fuerte el empty state.
- Menos banners, más acción.

### Rankings

Estado:

- La pantalla es legible, pero sigue explicando demasiado antes de competir.

Hallazgos P0:

- El estado bloqueado usa demasiadas capas:
  - hero
  - notice
  - tres stats
  - card de activación
  - card de tabla general
  - dos cards más abajo
- La acción principal existe, pero queda enterrada entre explicación y estructura.
- Copy correcto en tono general, pero demasiado explicativo para una pantalla que debería empujar ranking o activación.

Decisión:

- Segunda candidata clara a simplificación estructural.
- Tiene que pasar a una jerarquía mucho más corta.

### Profile

Estado:

- No existe `/profile` como página separada.
- La cuenta vive hoy dentro de dashboard y parte del shell.

Hallazgos:

- No hay un contrato UI propio de perfil todavía.
- El riesgo es seguir metiendo datos personales en dashboard hasta volverlo híbrido entre panel, perfil y estado de pago.

Decisión:

- No abrir frente nuevo ahora.
- Tratarlo como deuda de arquitectura de pantallas.

### Admin

Estado:

- Auditada por código.
- Sin sesión admin viva en esta pasada.

Hallazgos:

- La operación está clara, pero el lenguaje todavía mezcla producto y operación.
- `Qué sigue` es útil, pero sigue siendo una explicación larga al final.
- Debe consolidar sus propios patrones operativos y no heredar demasiado del lenguaje del jugador.

Decisión:

- Mantener separado del refactor de jugador.
- Revisarlo después de Dashboard/Matches/Rankings.
